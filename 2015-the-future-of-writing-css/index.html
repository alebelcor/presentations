Presentation is somewhat decoupled from HTML

Easy to learn, hard to master.

Sumed up into two main parts: selector, declarations

A selector is a pattern that will match one or more elements

A declaration is a particular style applied to the element(s) in the form of a property name, colon and a value

If it looks simple it's because it is. CSS is easy to learn, but hard to master

Why?

  • Scalability
  • Maintainability
  • Compatibility

Hard to write so it doesn't grow to be a monster. OOCSS, SMACSS, BEM and ITCSS

Hard to update or fix without creating side effects or regressions

As a developer, tons of quirks you have to be aware of for cross-browser compatibility

Most, if not all, preprocessors have all these features

Let's go over them quickly

Variables

Avoid repetition and allow for configuration.

$prefix: ns;
$blue: #00bcd4;

.#{$prefix}-header {
  background-color: $blue;
}
Outputs: 
.ns-header {
  background-color: #00bcd4;
}

They help us write less repetitive code

Whether is via changing colors, fonts, etc.

A repeated value should ideally be only changed in one place

Nesting

Visual hierarchy

.header {
  .nav {
    background-color: #00bcd4;
  }
}
Outputs: 
.header .nav {
  background-color: #00bcd4;
}

Provides a visual hierarchy when constructing selectors

Also helps us with selector repetition

Partials

Help organize code.

// _base.scss
.base { width: 980px; }

// main.scss
@import "base";

.main { margin: 0 auto; }
Outputs: 
.base { width: 980px; }

.main { margin: 0 auto; }

Partials are snippets of code you can include anywhere

Great way to organize code without the performance hit

Mixins

Reusable functions with output.

@mixin pull-left() {
  float: left;
}

.main {
  @include pull-left;
}
Outputs: 
.main {
  float: left;
}

A reusable function that will output one or more declarations

Useful when you need logic or repetitive declarations

Extends

Selector inheritance

.container {
  width: 980px;
}

.main {
  @extend .container;
}
Outputs: 
.container, .main {
  width: 980px;
}

A way of selector inheritance that will associate the declarations of one to another.

Operators

Do the math

h1 {
  font-size: (16px * 2);
}
Outputs: 
h1 {
  font-size: 32px;
}

You can add, subtract, multiply and divide

Problems

  • Nesting
  • Mixins
  • Extends
  • Logic inside styles
  • Monolithic

Preprocessors are great, but we can do better.

Nesting: a bad idea if nesting more than two levels deep. Creates long selectors and higher specificity

Mixins: say for browser prefixes, you have to maintain whatever the output is

Extends: becomes a bad idea really fast if you overextend. Creates long selectors and higher specificity

Logic inside styles: Logic should belong in a programming language, it makes CSS harder to read and to reason about

Monolithic: Means the code is huge, fixing bugs and making releases takes time and you're stuck if you don't like how a feature works

PostCSS

A tool for transforming CSS with JavaScript plugins.

Another CSS preprocessor? No, not quite

Call it a tool, transformer, postprocessor, transpiler or whatever. You write CSS and it will output transformed CSS

How does it work?

  • Parser
  • Stringifier

This makes it small compared against others.

It has two parts: a parser and a stringifier

The Parser takes in CSS and outputs an AST

The Stringifier takes in an AST and outputs CSS

How small?

Name Lines of code
PostCSS 2100~
Stylus 8000~
Ruby Sass 13000~
Less.js 13300~
LibSass 21900~

It's also very fast.

It's almost 4 times smaller than Stylus

It's at least 6 times smaller than Ruby Sass and Less.js

It's at least 10 times smaller than LibSass

How fast?

Name Time
PostCSS 36ms
LibSass 136ms
Less.js 160ms
Stylus 167ms
Ruby Sass 1084ms

Benchmark

Here's how it compares against other preprocessors in parsing, nested rules, mixins, variables and math

It's almost 4 times faster than LibSass

It's 4.5 times faster than Less.js and Stylus

It's 30 times faster than Ruby Sass

However

It doesn't have built-in variables, mixins or any of that.

PostCSS is a tool for transforming CSS with JavaScript plugins.

What's the deal? well, let's look at the definition again

That's what plugins are for. All features and transformations are done via plugins.

How did it work again?

  • Parser
  • Plugin(s)
  • Stringifier

Variables, nesting, partials, mixins, extends?

A plugin sits in the middle, it takes an AST and outputs a transformed AST

What about Sass' features? Can PostCSS do that?

Opacity for IE8

Manual way:

.thing {
  opacity: 0.9;
  -ms-filter: "progid:DXImageTransform.Microsoft.Alpha(Opacity=90)";
}

Sass way:

.thing {
  @include opacity(0.9);
}

IE8 requires a filter value.

Manual way: remember to type that in every time, error prone, tedious

Sass way: create a mixin. Better but what if you don't support IE8 anymore?

Opacity for IE8

PostCSS Opacity

.thing {
  opacity: 0.9;
}
Outputs: 
.thing {
  opacity: 0.9;
  -ms-filter: "progid:DXImageTransform.Microsoft.Alpha(Opacity=90)";
}

You just write regular CSS. You don't have to think about it

You don't have to remember to include a mixin

What if you don't support IE8 anymore? You just stop using the plugin.

I assume you have a package.json, gulpfile.js and that gulp is installed both globally and locally

A simple pipeline, no doing anything except copying a file

Next we'll install a plugin for Gulp to start using PostCSS

Adding nesting

npm install postcss-nested --save-dev
var gulp = require('gulp');
var postcss = require('gulp-postcss');

var nested = require('postcss-nested');

gulp.task('default', function () {
  return gulp.src('assets/stylesheets/style.css')
    .pipe(postcss([ nested ]))
    .pipe(gulp.dest('dist/'));
});

We'll install the plugin and add it the process

PostCSS exposes a function that takes an array of plugins and pipe the result forward

Adding variables

npm install postcss-simple-vars --save-dev
var gulp = require('gulp');
var postcss = require('gulp-postcss');

var nested = require('postcss-nested');
var vars = require('postcss-simple-vars');

gulp.task('default', function () {
  return gulp.src('assets/stylesheets/style.css')
    .pipe(postcss([ nested, vars ]))
    .pipe(gulp.dest('dist/'));
});

For most of the cases this is all you'll need to do, no need to configure anything

Let's look at a plugin that does require configuration

Adding prefixes

npm install autoprefixer-core --save-dev
var gulp = require('gulp');
var postcss = require('gulp-postcss');

var nested = require('postcss-nested');
var vars = require('postcss-simple-vars');
var autoprefixer = require('autoprefixer-core');

gulp.task('default', function () {
  return gulp.src('assets/stylesheets/style.css')
    .pipe(postcss([
      nested,
      vars,
      autoprefixer({ browsers: ['ie >= 8, last 2 versions'] })
    ]))
    .pipe(gulp.dest('dist/'));
});

You pass a configuration object to the exported function and that's it

Advantages for build

  • Easy to add/remove a plugin.
  • Requires little to no configuration.
  • No need for Grunt/Gulp wrapper plugins.

It's just a couple of lines in your build script.

Most plugins don't need to be configured and those that do accept a simple JS object.

JSHint, CSSLint, these all require a wrapper plugin to work with Grunt/Gulp

For the skeptics

You don't have to stop using preprocessors.

Preprocessors output CSS.

PostCSS transforms CSS.

  1. Write Sass.
  2. Sass outputs CSS.
  3. PostCSS transforms it.
